02

El proyecto

Stats de truco
entre amigos

  • Un grupo de ~20 personas juega torneos y partidos sueltos.
  • Cualquiera puede cargar resultados.
  • Un admin los aprueba y se publican en el dashboard.
  • El sistema calcula ELO, rachas, parejas, peor enemigo.
  • Caso real: dominio chico, modelado rico.

La base primaria

MongoDB Atlas

  • Replica set de 3 nodos en tier gratuito M0.
  • 5 colecciones modelan todo el dominio.
  • Comportamiento CP (consistencia > disponibilidad).
  • Write concern majority en cada operación.
  • JSON Schema validator activo en jugadores.

05

El modelo de datos

Las 5 colecciones

jugadores
24
  • username único
  • nombreCompleto
  • eloActual
  • activo, origen
partidos
20
  • equipoA / B / Ganador → ref
  • eloSnapshot EMB
  • fecha, tipoPartido, ronda
  • torneoId (opcional)
torneos
3
  • nombre, fecha, formato
  • equipos arrays bidim. → ref
  • equipoGanador
  • modalidad (2v2/3v3/1v1)
elo_historial
~80
  • jugadorId → ref
  • eloAnterior → eloNuevo
  • delta, fecha
  • partidoId, torneoId
pendientes
1-5
  • torneo EMB
  • partidos EMB ARR
  • estado: pendiente/aprobado
  • fechaEnvio, fechaResolucion
  • 3 decisiones de modelado: eloSnapshot embebido, equipos referenciados, pendientes embebido como árbol completo.
06

Cliente visual

MongoDB Compass

  • Cliente oficial para inspeccionar Atlas en vivo.
  • Vemos las 5 colecciones con su contenido real.
  • Aplicamos el JSON Schema validator desde mongosh integrado.
  • Verificamos los 9 índices ESR con explain().
  • Sirve para defender en vivo cualquier pregunta de modelo.
07

El driver oficial

PyMongo

  • Driver oficial de MongoDB para Python.
  • Soporta sesiones + transacciones ACID.
  • Auto-retry ante errores transitorios de red.
  • Habilita los 9 índices compuestos con regla ESR.
  • Aggregations declarativas y eficientes.
Driver oficial ACID ESR Aggregations

11

El punto más crítico

Transacción ACID

  • Cuando el admin aprueba un pendiente:
  • Se insertan jugadores nuevos, el torneo, partidos y elo_historial.
  • Se actualizan los eloActual y el estado del pendiente.
  • Todo dentro de una sesión Mongo.
  • Si algo falla → rollback completo. Todo o nada.
  • Probado con test automatizado contra Atlas real.
5 colecciones escritas with_transaction Tests de rollback
